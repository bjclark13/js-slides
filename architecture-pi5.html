### Back To Testing - We have done a great job of starting to implement automated testing in our processes - We have reported 40% code coverage already! - We know that we have had frustrations will failing tests
### Why Do Tests Fail Tests fail from two main reasons: 1. Our code does not work well 2. The test does not work well
### When code does not work well When our code does not work well, we simply need to fix it. This is the purpose of testing. We should not be releasing features when the tests tell us our code is not working
### When the test does not work well Tests are code too, so this means that sometimes we write poor tests. When this happens we can either 1. Fix the tests right away 2. Skip the test (by adding .skip around your spec) and create a bug ticket for *fixing the test functionality* Option number two allows you to not disrupt your development (people > process), but also not ignore the problem. If the test that is failing is for the feature you are currently working on, you should always do option number one. *Ignoring failing tests is not an option*
### Common Cypress Mistakes - Using async/await - Using the UI to log in - Relying on test data that is bound to change
### Common Cypress Fixes - Using async/await *Don't* - Using the UI to log in *Use cy.login instead* - Relying on test data that is bound to change *Use mocking and stubbing, shallow render, and/or right unit tests instead.* See https://docs.cypress.io/guides/references/best-practices.html
